# /wf-plan

既存ワークフローのステップバイステップ実装計画を作成する。

## 前提条件

- `docs/wf/<work-id>/02_SPEC.md` に仕様が完了済みであること
- `state.json` で仕様フェーズが完了済みであること

## ワークフロー

### ステップ 1: コンテキストの読み込み

1. `docs/wf/<work-id>/state.json` を読み、仕様が完了済みか確認
2. `docs/wf/<work-id>/01_KICKOFF.md` でゴールと制約を確認
3. `docs/wf/<work-id>/02_SPEC.md` で要件を確認
4. 仕様フェーズが未完了の場合、`/wf-spec` を先に実行するよう案内

### ステップ 2: 分析と設計

コードベースを調査して以下を決定する:
- 作成または変更が必要な具体的なファイル
- 正しい変更順序（依存関係を考慮）
- リスクと潜在的なロールバック戦略
- 各ステップの見積もり規模

### ステップ 3: 計画のドラフト

1. このスキルに同梱されている `templates/03_PLAN.md` を `docs/wf/<work-id>/03_PLAN.md` にコピー
2. 計画を記入:
   - **概要**: 高レベルなアプローチを1-2文で
   - **ステップ**: 各ステップは以下を満たすこと:
     - 独立して完了・検証できる小ささ
     - 依存関係順に並べる（基盤的な変更を先に）
     - どのファイルに触れるか明確
     - 具体的な完了基準を持つ
   - **進捗表**: ステップごとに1行、すべて "pending" で開始
   - **リスク**: 影響度、確率、緩和策を含むリスク一覧
   - **ロールバック計画**: 問題発生時の取り消し方法

### ステップのガイドライン

- 3-10ステップを目指す。10を超える場合は関連する変更をグループ化する。
- 各ステップは単独でテスト可能であること。
- 各ステップに「何を」（タスクリスト）と「なぜ」（目的）の両方を含める。
- ファイルパスは具体的に指定する（例: `src/utils/csv.ts`、「CSVモジュール」ではなく）。

### ステップ 4: ユーザーとのレビュー

計画を提示してフィードバックを求める。よくある調整:
- ステップの並び替え
- ステップの分割または統合
- ステップごとのスコープ調整
- リスクの追加/削除

### ステップ 5: 完了処理

1. `state.json` を更新:
   - `phases.plan.status` を `"completed"` に設定
   - `phases.plan.completed_at` を現在のタイムスタンプに設定
   - `phases.impl.total_steps` をステップ数に設定
   - `phase` を `"impl"` に設定
2. ユーザーに通知: 計画完了、`/wf-impl <work-id>` で次に進めることを伝える

## 重要な注意事項

- 計画書は日本語で作成する（テンプレートの言語に従う）。
- ステップは仕様の要件に明確にマッピングし、トレーサビリティを確保する。
- 計画はリビングドキュメントであり、`/wf-review` で問題が発見された場合に
  実装中に更新される可能性がある。
- `state.json` の全タイムスタンプはUTCで `YYYY-MM-DDTHH:MM:SSZ` 形式を使用する。
